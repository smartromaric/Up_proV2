# UpJunoo Pro — Intégration API back-office

> Document de référence (juin 2026) — base API : `https://api.upjunoo-dev.tech`  
> Source Swagger : `SWAGGER.md` · Code front : `src/core/api/links.ts`

---

## 1. Configuration front

| Variable | Rôle |
|----------|------|
| `NEXT_PUBLIC_API_URL` | Hôte API (sans `/v1` ni `/api/v2`) |
| `NEXT_PUBLIC_USE_MOCKS` | `true` → MSW + chemins `/api/v2/...` |
| `NEXT_PUBLIC_USE_REAL_AUTH` | `true` → auth + dashboard + carte live en `/v1/...` même si mocks actifs |
| `NEXT_PUBLIC_MAPBOX_ACCESS_TOKEN` | Carte live Mapbox (`pk....`) |

**Règle de routage HTTP** (`fetchClient.resolveApiUrl`) :

- Chemin commençant par `/v1/` → `{API_URL}/v1/...`
- Sinon → `{API_URL}/api/v2/...` (legacy MSW / ancien back-office)

**Headers envoyés (routes protégées)** :

```http
Authorization: Bearer <access_token>
Content-Type: application/json
Accept: application/json
X-Client-Type: back-office
X-Scope: platform | franchise | owner
X-Franchise-Id: <optionnel>
X-Owner-Id: <optionnel>
```

## 4. Écarts Swagger ↔ API réelle ↔ attentes front

### 4.1 Schémas absents ou vides dans `SWAGGER.md`

Le Swagger liste les paths mais **peu de schemas de réponse** pour :

- `GET /v1/admin/dashboard`
- `GET /v1/admin/live-map`
- `GET /v1/admin/orders`
- `GET /v1/admin/drivers`

→ Ce document et les fichiers `*.api.types.ts` du front servent de contrat de fait.

### 4.2 Auth — enveloppe réelle (OK côté front)

**Attendu front** (`AuthSession`) :

```json
{
  "token": "<jwt>",
  "refreshToken": "<string>",
  "user": {
    "id": "uuid",
    "name": "Dev Admin",
    "email": "dev.admin@upjunoo-dev.tech",
    "role": "admin",
    "scope": "platform",
    "permissions": ["ops.dashboard.view", "..."]
  }
}
```

**Réponse réelle** `POST /v1/auth/login` (extrait) — **non documentée telle quelle dans Swagger** :

```json
{
  "status": "ok",
  "role": "ADMIN",
  "userType": "ADMIN",
  "accessToken": "eyJ...",
  "refreshToken": "lqj3eedzjb3y",
  "expiresIn": 3600,
  "session": { "access_token": "eyJ...", "expires_in": 3600 },
  "profile": {
    "id": "1aed72f8-5ac4-4f68-a152-3705551006c5",

    "first_name": "Dev",
    "last_name": "Admin",
```

**Écarts** :

| Point | Swagger | Réel | Front |

### 4.5 Listes paginées (toutes pages tableau)

**Attendu front** (`Paginated<T>`) :

```json
{
  "data": [ { "...": "..." } ],
  "meta": {
    "total": 120,
    "per_page": 20,
    "current_page": 1,
    "last_page": 6
  }
}
```

**Query front** : `page`, `per_page`, `search`, `status`, `franchise_id`, `partner_id` (snake_case).

**À valider sur v1** : le backend utilise probablement `limit`/`offset` ou `page`/`limit` — **non aligné documenté**. Chaque migration liste devra inclure un adaptateur pagination.

### 4.6 Erreurs API

**Front** (`apiClient`) accepte :

```json
{ "message": "..." }
```

ou

```json
{ "error": { "code": "AUTH_REQUIRED", "message": "Missing bearer token" } }
```

Swagger référence `AUTH_REQUIRED` — **OK** après correctif client.

## 5. Ce que le front doit avoir (contrats stables)

Pour chaque domaine migré vers `/v1` :

1. **`links.ts`** — entrée `admin.v1.*`
2. **`*.api.types.ts`** — types bruts API
3. **`*.mapper.ts`** — conversion vers `@/shared/types`
4. **`*.service.ts`** — `useLegacyX()` + appel v1 si `useRealAuth || !useMocks`
5. **Handler MSW** `*/v1/admin/...` pour dev hors ligne (optionnel)
6. **IDs** — prévoir `string | number` partout (UUID)
7. **Filtres** — query `franchiseId` / `partnerId` en **UUID**, pas `franchise_id` numérique

## 8. Résumé pour le backend

Demandes de contrat pour accélérer le front :

1. **Schemas OpenAPI** complets pour dashboard, live-map, orders, drivers, withdrawals.
2. **`filterOptions`** (franchises / partenaires) sur dashboard et live-map.
3. **Pagination unifiée** : documenter `page` + `per_page` + shape `meta`.
4. **Positions GPS** chauffeurs sur live-map (ou endpoint dédié avec `lat`/`lng` récents).
5. **Permissions** utilisateur dans `GET /v1/auth/me` (ou endpoint rôles).
6. Aligner **URLs auth** documentées = URLs réelles (`password/forgot` vs `forgot-password`).
7. **Liste commandes** : champs alignés table front (`ref`, `from_label`, `to_label`, `client_name`, `driver_name`, `amount_fcfa`, `status`, `service`).
